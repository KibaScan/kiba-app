# Kiba Context Maintenance Guide — Supplemental

> **Why this exists:** Claude Code relies on SKILL.md, reference files, and DECISIONS.md to understand Kiba. When these files drift from reality, the agent works with wrong assumptions — leading to bugs, rework, and scoring regressions. This guide covers how to keep everything current.

---

## 1. What Drifts and Why It Matters

| File                              | What drifts                                        | Impact when stale                                              |
|-----------------------------------|----------------------------------------------------|----------------------------------------------------------------|
| **SKILL.md**                      | Test count, decision count, migration count, milestone phase, product count | Agent thinks fewer tests exist, misses new rules, works in wrong milestone scope |
| **references/scoring-details.md** | New scoring rules, weight changes, new bypasses, color system updates | Agent implements scoring against outdated math — silent regressions |
| **references/scoring-rules.md**   | New AAFCO thresholds, curve parameters, sub-nutrient weights, breed modifiers | Agent uses wrong thresholds or curves, produces incorrect scores |
| **references/ui-components.md**   | New screens, renamed components, removed components, new patterns | Agent creates duplicate components or references things that don't exist |
| **references/project-context.md** | New decisions, schema changes, new health conditions, milestone scope | Agent follows outdated schema or makes decisions already settled |
| **DECISIONS.md**                  | New decisions not appended, supersessions not marked | Agent relitigates settled questions or follows overridden decisions |
| **ROADMAP.md**                    | Completed milestones not marked, scope changes      | Agent works on completed features or scopes into future milestones |

## 2. SKILL.md — The Master Context File

This is loaded on every Claude Code session. Stale numbers here affect every interaction.

### What to audit

```
Prompt:
"Audit SKILL.md against the actual codebase. For each item below,
check the real value and update if it's changed:

1. Test count and suite count — run `npm test` and capture the summary
2. Decision count — count entries in DECISIONS.md, note the latest D-number
3. Superseded decisions — verify all supersession markers are present
   (currently: D-013 superseded by D-137, D-113 superseded by D-136 —
   are there new ones?)
4. Migration count — count migration files in the DB directory
5. Product count — query or check the data source
6. Current milestone phase — check ROADMAP.md for what's active
7. Non-negotiable rules — have any new rules been added via decisions
   that should be in the Non-Negotiable Rules list?
8. Schema Gotchas table — are there new columns, renamed fields, or
   new tables that would trip up an agent?
9. Common Pitfalls — have any new pitfalls been discovered since the
   last update?
10. What NOT to Build — has anything been added or removed from the
    rejection list?

Update SKILL.md with the correct values. Don't change the structure
or advice — only make the data current."
```

### Quick numbers check (lightweight, run anytime)

```
Prompt:
"Quick SKILL.md freshness check:
- Run the test suite and compare count to what SKILL.md says
- Count decisions in DECISIONS.md and compare
- Check current milestone in ROADMAP.md and compare
- Report what's drifted. Update SKILL.md if anything's off."
```

## 3. Reference Files — Scoring & Data Mapping

These are the files the agent reads on-demand before scoring or UI work. They were likely generated by Claude Code at some point, which means they reflect the codebase state *at the time they were written*.

### references/scoring-details.md

Contains: category-adaptive weights, DCM rules, allergen override, supplemental classification, score/severity color systems, regression targets.

```
Prompt:
"Audit references/scoring-details.md against the actual scoring
implementation in the codebase. For each section:

1. Category-adaptive weights — read the actual weight values from
   the scoring engine source code. Flag any that don't match the doc.
2. DCM D-137 rules — verify Rule 1 and Rule 2 parameters, mitigation
   values, and pulse detection logic against the implementation.
3. D-129 allergen override — verify dual-IQ scoring logic matches.
4. Supplemental classification (D-136/D-146) — verify 65/35/0 weights
   and topper/mixer classification logic.
5. Score color system — verify getScoreColor() ranges and hex values
   match what the doc says.
6. Severity colors — verify SEVERITY_COLORS from constants.ts matches.
7. Regression targets — verify Pure Balance = 62 and any other targets.
8. Check for any new scoring rules added via recent decisions that
   aren't documented here.

Update the doc to match reality. Flag anything that looks intentionally
changed vs accidentally drifted."
```

### references/scoring-rules.md

Contains: DMB conversion, AAFCO thresholds, trapezoidal curves, sub-nutrient weights, life stage/breed modifiers, carb estimation.

```
Prompt:
"Audit references/scoring-rules.md against the actual implementation:

1. DMB conversion formula — verify against the code
2. AAFCO thresholds — compare documented values to what the code uses
3. Trapezoidal curves — verify curve parameters (breakpoints, slopes)
4. Sub-nutrient weights — verify each weight value
5. Life stage modifiers — verify values and which life stages are handled
6. Breed modifiers — verify ±10 cap is documented, verify actual modifiers
7. Carb estimation formula — verify against implementation

For any values that don't match, update the doc to reflect the code.
The code is the source of truth — the doc should mirror it."
```

### references/ui-components.md

Contains: project structure, component inventory, screen architecture.

```
Prompt:
"Audit references/ui-components.md against the actual codebase:

1. Project structure — does the documented directory tree match reality?
   Flag any new directories, removed directories, or renamed paths.
2. Component inventory — list all components in the actual component
   directory. Flag any that exist in code but not in the doc, or
   in the doc but not in code.
3. Screen architecture — list all screens/routes. Flag any new screens,
   removed screens, or renamed screens.
4. Patterns — are there any new UI patterns being used in recent
   components that aren't documented?

Update the doc. Remove references to deleted components. Add new ones
with brief descriptions following the existing format."
```

### references/project-context.md

Contains: key decisions, schema, health conditions, cross-reactivity, milestone scope, rejections.

```
Prompt:
"Audit references/project-context.md against current state:

1. Key decisions — cross-reference with DECISIONS.md. Are recent
   decisions (after whatever D-number this doc was last updated to)
   reflected here?
2. Schema — compare documented schema to actual Supabase schema.
   Flag new tables, new columns, changed types, new RLS policies.
3. Health conditions — verify the list matches what the code handles.
4. Cross-reactivity — verify mappings against the implementation.
5. Milestone scope — verify this matches ROADMAP.md.
6. Rejections — verify the 'What NOT to Build' list is current.

Update to match reality."
```

## 4. When to Run These Audits

| Trigger                          | What to audit                                     | Prompt to use                        |
|----------------------------------|---------------------------------------------------|--------------------------------------|
| **Closing a milestone**          | Everything — SKILL.md, all references, DECISIONS  | Full audit (all prompts above)       |
| **After a scoring change**       | scoring-details.md, scoring-rules.md, SKILL.md    | Scoring audits + quick numbers check |
| **After adding 5+ decisions**    | SKILL.md (decision count), project-context.md     | SKILL.md audit + project-context     |
| **After adding new screens/components** | ui-components.md                            | UI components audit                  |
| **After schema migration**       | SKILL.md (migration count), project-context.md    | Schema sections of each              |
| **Monthly (regardless)**         | SKILL.md quick numbers check                      | Quick numbers check prompt           |
| **Starting a new milestone**     | ROADMAP.md, SKILL.md (current phase)              | Quick numbers check prompt           |

## 5. Slash Command: /audit-context

Add this to `.claude/commands/audit-context.md` to make the full audit a single command:

```markdown
Run a full context freshness audit for Kiba. Check each file
against the actual codebase and fix any drift:

1. SKILL.md — test count, decision count, migration count,
   product count, milestone phase, supersession markers,
   non-negotiable rules, schema gotchas, common pitfalls
2. references/scoring-details.md — weights, DCM rules, allergen
   override, supplemental classification, color systems, regression targets
3. references/scoring-rules.md — DMB, AAFCO thresholds, curves,
   sub-nutrient weights, life stage/breed modifiers, carb estimation
4. references/ui-components.md — directory tree, component inventory,
   screen list, UI patterns
5. references/project-context.md — decisions, schema, health conditions,
   cross-reactivity, milestone scope, rejections
6. DECISIONS.md — verify latest D-number, check for unmarked supersessions
7. ROADMAP.md — verify current milestone, mark completed ones

For each file, report:
- What changed (with before → after values)
- What was already accurate
- Anything suspicious that needs human review

Do NOT change advice, structure, or tone — only update data to
match reality.
```

## 7. Preventing Drift at the Source

The best approach isn't auditing after drift — it's preventing drift in the first place. Add these steps to your existing workflows:

### Add to milestone-close.md

```markdown
# Append these steps to the existing milestone-close slash command:

9.  Run /check-numbers and update SKILL.md with current values
10. Check if any new decisions since last milestone should be added
    to SKILL.md non-negotiable rules or schema gotchas
11. Verify all reference files (scoring-details, scoring-rules,
    ui-components, project-context) are current — run /audit-context
    if the milestone involved scoring changes, schema changes, or
    new screens
```

### Add to handoff.md

```markdown
# Append this to the existing handoff slash command:

- If this session added new decisions: note the new D-numbers and
  flag if SKILL.md decision count needs updating
- If this session changed scoring logic: flag that scoring reference
  files may need auditing next session
- If this session added migrations or new tables: flag that SKILL.md
  migration count and schema gotchas may need updating
```

### After any scoring change

```
Prompt:
"I just changed scoring logic. Do the following:
1. Run the full test suite — report count (update SKILL.md if it changed)
2. Verify Pure Balance = 62
3. Check if references/scoring-details.md needs updating for this change
4. Check if references/scoring-rules.md needs updating for this change
5. If either reference file is now stale, update it"
```

## 8. The Self-Documenting Decision Pattern

When you make a new decision in Claude Code, have the agent update all affected files in the same session:

```
Prompt:
"We just made decision D-[N]: [description].
1. Append it to DECISIONS.md following the existing format
2. Update SKILL.md decision count
3. If this decision supersedes an earlier one, add supersession
   markers to both the old and new entries
4. If this decision affects scoring, update references/scoring-details.md
5. If this decision affects schema, update references/project-context.md
6. If this decision adds a non-negotiable rule, add it to the
   Non-Negotiable Rules section of SKILL.md
7. If this decision adds a common pitfall, add it to Common Pitfalls
8. If this decision adds to the 'What NOT to Build' list, add it"
```

## Quick Reference

```
AFTER SCORING CHANGE     →  Verify Pure Balance, audit scoring references
AFTER NEW DECISION       →  Use self-documenting decision pattern
AFTER SCHEMA MIGRATION   →  Update SKILL.md migration count + schema gotchas
AFTER NEW SCREENS        →  Audit ui-components.md
CLOSING A MILESTONE      →  Run /audit-context (full audit)
BETWEEN MILESTONES       →  Run /check-numbers (lightweight)
MONTHLY (MINIMUM)        →  Run /check-numbers
```

---

*The underlying principle: reference files are only valuable if they're current. A reference file that says 558 tests when you have 925 isn't just outdated — it's actively misleading every agent session. Budget 10 minutes at milestone close to run the audit. It pays for itself in the first session that doesn't hallucinate based on stale data.*
